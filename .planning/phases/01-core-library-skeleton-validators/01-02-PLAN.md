  <name>Cello range and bar-duration validators</name>
  <files>core/engine/validators.py, tests/test_validators.py</files>
  <behavior>
    - `validate_pitch("C2")` returns `None` (no raise) â€” MIDI 36, exactly at the floor.
    - `validate_pitch("D5")` returns `None` (no raise) â€” MIDI 74, exactly at the default ceiling.
    - `validate_pitch("B1")` raises `ValueError` â€” MIDI 35, one semitone below the floor.
    - `validate_pitch("G5")` raises `ValueError` when `extended=False` (default) â€” MIDI 79, above the D5 default cap but below the C6 extended cap.
    - `validate_pitch("G5", extended=True)` returns `None` (no raise) â€” MIDI 79, within the C6 (MIDI 84) extended cap.
    - `validate_pitch("C6", extended=True)` returns `None` (no raise) â€” MIDI 84, exactly at the extended ceiling.
    - `validate_pitch("C#3")` returns `None` (no raise) â€” MIDI 49, proves accidentals resolve via `Pitch.midi`, not lexical string comparison (RESEARCH.md Pitfall 4).
    - `validate_pitch("Bb2")` returns `None` (no raise) â€” MIDI 46, another accidental edge case from the actual preset data.
    - The raised `ValueError` message contains both the offending pitch name and the MIDI number (human-readable per CONTEXT.md).
    - `validate_bar_duration([1.0, 1.0, 1.0, 1.0], "4/4")` returns `None` (no raise) â€” sums to 4.0, matches `TimeSignature("4/4").barDuration.quarterLength`.
    - `validate_bar_duration([1.0, 0.5, 0.5, 1.0, 1.0], "3/4")` raises `ValueError` â€” sums to 4.0, mismatches 3/4's 3.0 bar length (proves the function does NOT hardcode 4.0 as "the" bar length; it must reject a 4.0 sum against a 3/4 meter).
    - `validate_bar_duration([1.0, 1.0, 1.0], "3/4")` returns `None` (no raise) â€” sums to 3.0, matches 3/4's bar length exactly.
    - `validate_bar_duration([0.5, 0.5, 1.0, 0.5, 0.5, 0.5, 0.5], "4/4")` returns `None` (no raise) â€” this is the exact `ritual_tribal`/`sexy_duet` cello rhythm from the real preset data (RESEARCH.md/PATTERNS.md), sums to 4.0.
    - `validate_bar_duration([1.0, 1.0, 1.0, 1.0], "3/4")` raises `ValueError` with a message containing both the actual sum and the expected bar length for the given meter.
  </behavior>
  <implementation>
    Implement per the verified reference in 01-PATTERNS.md/01-RESEARCH.md: `from music21 import meter, pitch` (narrow import â€” do not import unused submodules like `instrument`/`clef`/`tempo` that the CLI scripts use for score-building). Define module-level constants `CELLO_MIN_MIDI = 36`, `CELLO_MAX_MIDI_DEFAULT = 74`, `CELLO_MAX_MIDI_EXTENDED = 84` with a one-line comment noting the note names they correspond to (C2/D5/C6) since the bare integers are non-obvious without it. `validate_pitch(pitch_name: str, extended: bool = False) -> None` builds `p = pitch.Pitch(pitch_name)`, selects `max_midi` based on `extended`, and raises `ValueError(f"Pitch {pitch_name} (MIDI {p.midi}) is outside playable cello range ({CELLO_MIN_MIDI}-{max_midi}).")` when `p.midi` is outside `[CELLO_MIN_MIDI, max_midi]`. `validate_bar_duration(rhythm: list[float], meter_signature: str, tolerance: float = 1e-9) -> None` builds `ts = meter.TimeSignature(meter_signature)`, computes `expected_ql = ts.barDuration.quarterLength`, `actual_ql = sum(rhythm)`, and raises `ValueError(f"Bar duration {actual_ql} != {expected_ql} for meter {meter_signature}.")` when `abs(actual_ql - expected_ql) > tolerance`. No `__post_init__` hooks, no custom exception classes â€” plain functions raising stdlib `ValueError`, matching the zero-precedent-for-custom-exceptions convention across the whole codebase (01-PATTERNS.md "Error handling" section).
  </implementation>
</feature>
