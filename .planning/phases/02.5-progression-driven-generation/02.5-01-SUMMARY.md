**New `core/engine/progression.py` parses free-text chord progressions ("Am F C G", including extended qualities like F#m7/Gsus4/Cmaj7/Bdim) via pychord 0.2.8 into ordered chord objects with named bad-token errors, and `core/engine/loop_engine.py` gains a parallel chord-tone-to-cello-register-mapping path (`build_progression_score`/`generate_variant_from_progression`) that turns any parsed progression + existing mood preset into a validated, monophonic, C2-D5 cello Score with a fully populated per-bar `GenerationTrace`.**

## Accomplishments

- `parse_progression(text)` splits on whitespace, parses each token via `pychord.Chord`, and returns an ordered list of `ParsedChord(name, components)` -- unrecognized tokens raise `ValueError` naming the exact bad token verbatim (e.g. `"Xz9"` appears in the message), matching the codebase-wide bare-`ValueError` convention (no custom exception classes)
- Extended chord qualities parse correctly: `"F#m7"`, `"Gsus4"`, `"Cmaj7"`, `"Bdim"` all resolve to their correct component tones -- required normalizing `"maj7"` to pychord's own `"M7"` spelling first (see Deviations)
- Empty/whitespace-only input raises a clear `ValueError`; duplicate/surrounding whitespace is tolerated and produces identical parse results
- `build_progression_score(chords, preset, seed=None)` builds a music21 Score with exactly one bar per chord, reusing the preset only for its rhythm/tempo/meter/velocity strategy (the *when*) while a new chord-tone -> register-mapping step decides *which* pitch, within C2-D5 (confirmed C2 == MIDI 36 == `CELLO_MIN_MIDI`, D5 == MIDI 74 == `CELLO_MAX_MIDI_DEFAULT` -- the plan's stated range maps exactly onto the existing validator constants)
- Register mapping favors root/fifth chord tones in the low register (octaves 2-3), other chord tones default to mid register (octaves 3-4), and once a previous note exists in the line, the closest in-range candidate within an octave leap is preferred over the register bias -- keeping voice-leading smooth. Stress-tested across all 4 solo presets x 5 progressions (including extended-quality chords) x 5 seeds: zero validator failures, zero leaps larger than an octave
- Every generated bar uses only that bar's chord's component tones (verified directly against `ParsedChord.components`), output is strictly monophonic (no two notes share an offset within a measure), and every generated bar passes `validate_pitch`/`validate_bar_duration`
- `generate_variant_from_progression()` populates a fully non-null `GenerationTrace` per call: `pattern_strategy="progression_driven_register_mapped"`, `register_choices` and `chord_tones_used` each with exactly one entry per bar
- Same seed + same progression + same mood reproduces byte-identical MIDI output (verified via `streamToMidiFile().writestr()` comparison); different-seed runs are free to differ
- The pre-existing preset-verbatim path (`build_score`/`generate_variant`) is completely untouched -- all 18 pre-existing `tests/test_loop_engine.py` tests plus the full 80-test suite (including both golden regression tests: byte-identical MIDI and normalized-MusicXML) remain green

## Decisions Made

- Normalized chord-quality aliases (`maj7`->`M7`, `min`->`m`, `min7`->`m7`, `maj9`->`M9`, `-7`->`m7`) before calling `pychord.Chord()`, discovered necessary by directly inspecting pychord 0.2.8's `QUALITY_DICT` (only recognizes `M7`, raises on `maj7`) rather than assuming API compatibility from the tech-stack doc's example usage

- One bar per chord (not multi-bar-per-chord) -- simplest mapping satisfying ROADMAP SC2's wording, keeps per-bar continuity state easy to test
- Register-choice priority: voice-leading continuity (closest in-range candidate to the previous note) overrides the root/fifth-low-register bias whenever a previous note exists; only the progression's very first note falls back to the pure register-bias pool

## One-Paragraph Pitch-Mapping Judgment-Call Summary (for human ear-check)

The register-mapping algorithm makes several judgment calls with no single "correct" answer musically, so a human ear-check is recommended before treating this as production-quality voicing: (1) root and fifth chord tones are biased toward octaves 2-3 (low register), while thirds/sevenths/extensions default to octaves 3-4 (mid register) -- a simple, defensible heuristic for cello-idiomatic ostinato lines, but not derived from any specific voice-leading theory beyond "keep the bass functional tones low, let color tones sit above"; (2) whenever a previous note exists in the line, the closest in-range candidate pitch (by absolute MIDI distance, ties broken toward the lower octave) is chosen instead, meaning the register bias is really only a tiebreak/starting-point rule and the actual output is dominated by "stay close to where you just were" -- this avoids large leaps (verified: zero leaps exceed an octave across a 100-combination stress test spanning all 4 solo presets, 5 progressions including extended qualities, and 5 seeds) but does not implement genuine functional voice-leading (e.g. no preference for common tones between adjacent chords, no resolution tendencies for tritones/leading tones); (3) when a chord has more than the rhythm's note-count worth of distinct tones, or fewer, tones simply cycle (`chord.components[i % len(chord.components)]`) rather than applying any smarter distribution -- for a 4-tone chord (e.g. `Cmaj7`) against an 8-note rhythm this produces a clean two-cycle pattern, but for oddly-sized rhythm/chord-size combinations the cycling could feel arbitrary to a trained ear. None of this breaks correctness (every note still passes `validate_pitch`, stays in C2-D5, and never repeats an offset within a bar), but "sounds good" and "is technically valid" are different bars, and this phase only guarantees the latter.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Normalized chord-quality aliases before calling pychord**
- **Found during:** Task 1 (writing the failing test for extended qualities, before any implementation)
- **Issue:** The plan's `<behavior>` explicitly requires `"Cmaj7"` to parse successfully. Direct inspection of pychord 0.2.8's `QUALITY_DICT` showed it only recognizes `"M7"` for a major-seventh chord, not `"maj7"` -- `Chord("Cmaj7")` raises `ValueError: Invalid chord Cmaj7: Unknown quality maj7`. Without a fix, the plan's own required test case would be unsatisfiable with the pinned library version as-is.
- **Fix:** Added a `_normalize_token()` step in `core/engine/progression.py` that rewrites a small set of common lead-sheet quality aliases (`maj7`->`M7`, `min`->`m`, `min7`->`m7`, `maj9`->`M9`, `-7`->`m7`) to pychord's own spellings before parsing, while leaving the root note and any unrecognized suffix untouched so pychord's own error message (naming the *original* token) still surfaces correctly for genuinely bad input.
- **Files modified:** `core/engine/progression.py`
- **Verification:** `test_parse_progression_with_extended_qualities` (asserts `Cmaj7` -> `["C", "E", "G", "B"]`) passes; `test_parse_progression_unrecognized_token_names_bad_token_in_message` confirms the original (non-aliased) bad token still appears verbatim in the error for genuinely invalid input
- **Committed in:** `9006a26` (Task 1 GREEN commit)
