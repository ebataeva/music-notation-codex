# Phase 02.5: Code Review Report

**Reviewed:** 2026-07-05
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Reviewed the new progression-driven generation code (diff from dc37979): `parse_progression` in `core/engine/progression.py` and the chord-tone → cello-register mapping (`_all_candidate_pitches`, `_choose_register_for_chord_tone`, `_register_map_chord`, `build_progression_score`, `generate_variant_from_progression`) plus trace population in `core/engine/loop_engine.py`. All 80 tests pass. Every finding below was verified empirically by executing the code with the project venv, not just by reading it. Phase-02 findings already filed in `02-REVIEW.md` are not re-reported.

One BLOCKER: parsing a slash chord (e.g. `C/G`) through the public `parse_progression` API **permanently corrupts pychord's process-global quality cache** — every subsequent chord of that quality in the same process returns rotated (or, for sevenths, duplicated) components, silently breaking root/fifth detection and traces for the rest of the Streamlit session. The register-mapping algorithm also has an emergent defect: the "closest to previous note" rule always beats the documented low-register bias, so every generated line monotonically climbs to the top of the cello range (D5) within ~2 bars and parks there.

## Critical Issues

### CR-01: Slash-chord input permanently corrupts pychord's shared quality cache — all later chords in the process return wrong components

**File:** `core/engine/progression.py:72,77` (exposure), `core/engine/loop_engine.py:364-365,370` (consumption)
**Issue:** pychord 0.2.8 caches `Quality` objects globally, and building/parsing a slash chord **mutates the cached quality in place**. Verified empirically in a single process:

```
Chord("Bb").components()   -> ['Bb', 'D', 'F']        # before
Chord("C/G").components()  -> ['G', 'C', 'E']
Chord("Bb").components()   -> ['F', 'Bb', 'D']        # after: rotated, root no longer first
Chord("C").components()    -> ['G', 'C', 'E']         # every major chord now rotated
Chord("Am7").components()  -> ['G','A','C','E','G']   # after Am7/G: 5 components, duplicate G
```

`parse_progression` accepts slash tokens today (`C/G` parses successfully), so a single user input like "C/G F C G" in the long-lived Streamlit process poisons all subsequent progressions — verified: a later, slash-free `parse_progression("C F G Am")` returns `C -> ['G','C','E']`, `F -> ['C','F','A']`, `G -> ['D','G','B']`. Downstream, `_register_map_chord` (loop_engine.py:364-365) assumes `components[0]` is the root and `components[2]` is the fifth; after contamination the low-register bias is applied to the wrong tones, `i % len(components)` cycles a corrupted (possibly 5-element, duplicated) tone list, and `chord_tones_used` in the trace records the wrong tones in the wrong order — for every generation until process restart. This is user-input-triggered, process-global, irreversible state corruption producing incorrect output.
**Fix:** Reject slash tokens explicitly in `parse_progression` until slash-chord support is designed (this also resolves WR-03):
```python
for token in tokens:
    if "/" in token:
        raise ValueError(
            f"Slash chords are not supported yet: {token!r} in progression {text!r}."
        )
```
If slash chords must be supported, do not call pychord's slash path at all — parse root and bass separately and build components from the non-slash chord name.

## Warnings

### WR-01: Register mapping monotonically climbs to the top of the range and stays there — low-register bias is dead code after the first note

**File:** `core/engine/loop_engine.py:335-341` (closest-wins rule), `290-292` (contradicted comment)
**Issue:** `_choose_register_for_chord_tone` returns the leap-constrained candidate *closest* to the previous note whenever a previous note exists — i.e. for every note except the very first of the piece. When chord tones cycle upward (root → third → fifth → root, as `_register_map_chord` does), the ascending neighbor is always strictly closer than the descending one (3-5 semitones up vs 7-9 down), so the line ratchets upward until it hits the range ceiling. Verified: `Bb Eb Gm F` with seed 42 produces bar 1 `B-2 D3 F3 B-3 D4 F4 B-4 D5` (a two-octave climb in one bar), then bars 2-4 all classify "high register"; a 32-bar `Am F C G` loop ends permanently oscillating at MIDI 67/71/74 (top of range). The `is_root_or_fifth` low-register branch (lines 343-345) is reachable only for note 1 of the entire piece. This contradicts the module comment at lines 290-292 ("preferring the lowest register that satisfies root/fifth preference and the max-leap constraint") — the implementation prefers *closest*, not *lowest-satisfying* — and produces high-register-only output for a cello loop coach regardless of preset mood.
**Fix:** Apply the register bias *within* the leap-constrained set instead of short-circuiting past it:
```python
if previous_pitch is not None:
    within_leap = [c for c in candidates if abs(c.midi - previous_pitch.midi) <= 12]
    if within_leap:
        candidates = within_leap  # constrain, don't return
if is_root_or_fifth:
    pool = [c for c in candidates if c.octave <= 3] or candidates
else:
    pool = [c for c in candidates if c.octave in (3, 4)] or candidates
return rng.choice(pool)
```

### WR-02: Duet-only preset through the new progression path fails with the same misleading "Rhythm is empty" error that WR-06 fixed for `generate_variant`

**File:** `core/engine/loop_engine.py:381-402,435-450`
**Issue:** Phase-02 review WR-06 established that public entry points must not leak the internal `"Rhythm is empty for meter 4/4."` message for duet-only presets, and `generate_variant` (line 107) got the guard. The new public entry points `generate_variant_from_progression` / `build_progression_score` did not: verified `generate_variant_from_progression(parse_progression("Am F C G"), get_preset("sexy_duet"))` raises `ValueError: Rhythm is empty for meter 4/4.` — an error describing internal data representation, not the caller's mistake. The progression path depends on `preset.rhythm`, not `preset.bars`, so the existing guard pattern needs mirroring on the rhythm field.
**Fix:** At the top of `generate_variant_from_progression` (and/or `build_progression_score`):
```python
if not preset.rhythm:
    raise ValueError(
        f"Preset {preset.name!r} has no solo rhythm (duet-only preset); "
        f"choose one of: {', '.join(list_solo_presets())}."
    )
```

### WR-03: Quality-alias normalization is exact-match only — `Cmaj7/G` fails while `Cmaj7` succeeds

**File:** `core/engine/progression.py:41-50`
**Issue:** `_normalize_token` looks up the *entire* post-root remainder in `_QUALITY_ALIASES`. Any recognized alias followed by additional syntax misses the map: verified `parse_progression("Cmaj7/G")` raises `Unknown quality maj7` even though pychord natively parses `CM7/G` and the module's stated purpose is making `maj7` spellings work. The failure message ("Unknown quality maj7") is also confusing because bare `Cmaj7` succeeds. If CR-01's fix rejects slash tokens outright, this specific repro becomes a clear "slash chords not supported" error and this finding is resolved for free; if slash chords are kept, split the token on `/` before alias lookup and normalize only the quality segment.
**Fix:** Prefer the CR-01 slash rejection. Otherwise:
```python
quality_part, _, bass_part = rest.partition("/")
normalized = _QUALITY_ALIASES.get(quality_part, quality_part)
return root + normalized + (f"/{bass_part}" if bass_part else "")
```

## Info

### IN-01: `chord_tones_used` trace field has inconsistent semantics between the two generation paths

**File:** `core/engine/loop_engine.py:125` vs `core/engine/loop_engine.py:461`
**Issue:** `generate_variant` stores concrete octave-bearing pitches per bar (e.g. `["C2", "C2", ...]`, asserted by `test_generate_variant_chord_tones_used_matches_bars`), while `generate_variant_from_progression` stores octave-less pitch classes (e.g. `["A", "C", "E"]`). A downstream consumer (Phase 3 TheoryExplainer) reading `GenerationTrace.chord_tones_used` gets two different shapes under one field with no discriminator other than `pattern_strategy`.
**Fix:** Document the per-strategy semantics on `GenerationTrace.chord_tones_used` in `core/models.py`, or normalize both paths to pitch classes.

### IN-02: pychord's enharmonic spellings leak into generated notation (Gm renders A#, not Bb)

**File:** `core/engine/loop_engine.py:295-304`
**Issue:** pychord spells some qualities with sharps regardless of key context: verified `Chord("Gm").components() -> ['G', 'A#', 'D']`, and the generated score for a `Bb Eb Gm F` progression contains `A#4` in the G-minor bar. Playback is identical, but for an app whose core value is teaching-quality notation, A# in a flat-key context is a wrong spelling that OSMD will render verbatim in later phases.
**Fix:** After choosing the concrete pitch, respell against the preset key (music21: `p.getEnharmonic()` selection, or `p.simplifyEnharmonic()` with key context) — or note it as a known Phase-3 concern.

### IN-03: Fifth detection assumes triadic ordering — power chords' fifth is misclassified

**File:** `core/engine/loop_engine.py:364-365`
**Issue:** `fifth = chord.components[2] if len(chord.components) >= 3 else None`. For dyads like `C5` (components `['C', 'G']`, verified), the fifth is at index 1, so G gets the mid-register bias intended for thirds instead of the root/fifth low bias. Index 2 also isn't the fifth for every ≥3-tone quality pychord emits (and is meaningless post-CR-01 contamination). Only affects register bias, not correctness of tones.
**Fix:** Detect the fifth by interval from the root (7 semitones mod 12) rather than by position:
```python
root_pc = pitch.Pitch(root).pitchClass
fifth = next((c for c in chord.components[1:]
              if (pitch.Pitch(c).pitchClass - root_pc) % 12 in (6, 7, 8)), None)
```

### IN-04: Test suite never exercises flats or slash chords through the engine — the gap that let CR-01/WR-01 through

**File:** `tests/test_progression.py`, `tests/test_loop_engine.py:193-299`
**Issue:** Every progression-path test uses `Am F C G` (all naturals, no slash, no flats). The chord-tones assertion at `tests/test_loop_engine.py:238` carries a `.replace("b", "-")` dance for flats that no test input ever triggers, and its `or n.pitch.name in chord.components` clause is dead for flat spellings (music21 `E-` never equals pychord `Eb`). A single test parsing `"C/G F"` twice, or a flat-key progression, would have caught CR-01's contamination and IN-02's spelling. The leap-constraint test also pins one seed/progression only.
**Fix:** Add tests for (a) a flat-key progression end-to-end, (b) `parse_progression` called after a slash-chord parse returning root-first components (or the slash rejection error once CR-01 is fixed), (c) leap constraint over ≥16 bars.
