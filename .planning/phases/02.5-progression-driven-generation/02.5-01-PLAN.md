# Phase 2.5: Progression-Driven Generation — Agentic Execution Plan

**Scope:** implement chord-progression parsing → chord-tone extraction → cello-register mapping → rhythm-strategy application → validated bar generation, per ROADMAP.md Phase 2.5.

**Depends on:** Phase 2 (LoopEngine + ExportEngine) must be complete and its golden baseline green before starting.

---

## Autonomy contract

Run this plan end-to-end without stopping for confirmation, EXCEPT at the escalation triggers listed below. This means: write code → run tests → read failures → fix → re-run, repeated, without asking permission for each step.

**Iteration limit:** max 3 fix attempts per failing test. On the 4th failure of the same test, stop and report — do not keep guessing.

**Blast radius limit:** touch only `core/engine/`, `core/theory/` (if chord-tone logic belongs there), and `tests/` for this phase. Do not modify `core/models.py`, `core/presets/`, or anything in `scripts/` — those belong to other phases/waves.

---

## Steps (execute in order)

1. Add `pychord` to `requirements.txt`, pinned. Confirm it installs cleanly.
2. Write the progression parser: `"Am F C G"` → list of chord objects. Unrecognized token → raise a clear error naming the exact bad token (not a generic parse failure).
3. Write chord-tone → cello-register mapping: given a chord's tones, produce pitches within C2–D5, monophonic, idiomatic for cello (favor root/fifth in low register, avoid awkward leaps — use existing MoodPreset rhythm/pattern strategies as the source of *when* notes land; this step only decides *which* pitch).
4. Wire mapped pitches into `LoopEngine.build_score()` so an arbitrary parsed progression + mood produces a Score.
5. Populate `GenerationTrace` per bar: which chord tones were chosen, register decision, pattern strategy used. `GenerationTrace` in `core/models.py` already has the needed fields (`pattern_strategy`, `register_choices`, `chord_tones_used`, `voice_leading_steps`, `seed`) — construct instances, do not edit `models.py`.
6. Write unit tests for: valid progression parses correctly; invalid token raises named error; generated bars pass `validate_pitch` and `validate_bar_duration`; chord tones used match the input progression; trace is populated per bar.
7. Run `pytest tests/` in full — not just the new tests — to confirm no regression against the Phase 1/2 golden baseline.

---

## Escalation triggers (stop and ask instead of proceeding)

- A design decision isn't determined by the roadmap or existing code (e.g. how to voice a chord with more than 4 tones in a monophonic line — there's no single obviously-correct answer).
- Same test fails 4 times in a row.
- Fixing this phase would require changing a Phase 1/2 file outside the allowed blast radius.
- `pychord` can't represent a chord type the user is likely to type (e.g. slash chords, sus chords) — decide scope narrowing vs. custom parsing is itself a scope decision, not a bug fix.

---

## Definition of done

All four Success Criteria from ROADMAP.md Phase 2.5 pass, `pytest tests/` is fully green, and a one-paragraph summary of any judgment calls made during pitch-mapping is included in the final report (since this is the phase most likely to need a human ear-check afterward).
