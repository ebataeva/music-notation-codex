# Phase 02.5 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail.

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| User chord text → parser | Free-form user input crosses into pychord parsing; malformed tokens must fail closed with `ValueError`, never crash with a raw library exception | Untrusted free-text chord tokens |
| Parsed chords → engine | Chord tones flow into register mapping and score assembly; validators (D-06) gate every generated pitch/bar | Structured `ParsedChord` objects |

**Audit notes (2026-07-05):**
- T-02.5-01: empirical fuzz of 16 malformed tokens (incl. `Xz9`, `H7`, `Cmaj7/G`, control chars, 500-char token) — only `ValueError` escaped, offending token named in every message; no raw pychord exception reached the caller.
- T-02.5-02: 65-chord progression empirically raises `ValueError` on both entry points before Score construction. Scope caveat (non-blocking): `parse_progression` has no token-count cap, so a long string is fully parsed before the bar-count guard rejects it — parsing is linear/cheap and the declared mitigation ("guard raises before Score construction") is exactly what the code does.
- T-02.5-03: promoted from an unregistered flag found during code verification — the CR-01 slash-chord global-cache corruption is already mitigated by up-front `/`-token rejection (fuzz confirmed `Cmaj7/G` fails closed).
- All verification was read-only plus two ephemeral runtime probes; no implementation files were touched.
