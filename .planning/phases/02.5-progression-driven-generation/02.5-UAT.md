## Tests

### 1. Basic progression parsing
expected: |
  `parse_progression("Am F C G")` returns 4 ParsedChord entries in input order:
  Am -> [A, C, E], F -> [F, A, C], C -> [C, E, G], G -> [G, B, D].
result: pass

### 2. Extended chord qualities
expected: |
  `parse_progression("F#m7 Gsus4 Cmaj7 Bdim")` parses all 4 tokens successfully,
  including the maj7->M7 alias normalization, with correct component tones for each.
result: pass

### 3. Bad-token and empty-input errors
expected: |
  `parse_progression("Am Xz9 C")` raises ValueError naming "Xz9" verbatim.
  `parse_progression("")` and `parse_progression("   ")` both raise a clear ValueError.
result: pass

### 4. Generated Score validity
expected: |
  `build_progression_score(["Am","F","C","G"], preset)` produces exactly 4 bars (one per
  chord), every note within C2-D5, strictly monophonic (no offset collisions within a
  bar), and every bar uses only that bar's chord's component tones.
result: pass

### 5. GenerationTrace population
expected: |
  `generate_variant_from_progression(...)` returns a GenerationTrace with
  pattern_strategy == "progression_driven_register_mapped", and register_choices /
  chord_tones_used each containing exactly one non-null entry per bar.
result: pass

### 6. Seed reproducibility
expected: |
  Same seed + same progression + same preset -> byte-identical MIDI output across two
  separate calls. A different seed produces different output.
result: pass

### 7. Full suite regression (preset-verbatim path untouched)
expected: |
  Full `pytest tests/` run is green (88/88), including both golden regression tests
  (byte-identical MIDI + normalized-MusicXML). Known IN-04 side effect (volatile
  encoding-date/id rewrites in tracked scores/musicxml/*.musicxml) discarded via
  targeted `git checkout -- scores/` afterward, working tree left clean.
result: pass
