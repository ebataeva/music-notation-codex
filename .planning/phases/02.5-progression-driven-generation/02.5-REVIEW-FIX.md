## Fixed Issues

### CR-01: Slash-chord input permanently corrupts pychord's shared quality cache

**Files modified:** `core/engine/progression.py`, `tests/test_progression.py`
**Commit:** 567e192
**Applied fix:** `parse_progression` now rejects any token containing `/` with a clear "Slash chords are not supported yet" ValueError *before* the token ever reaches pychord, so the process-global quality cache can no longer be mutated by user input. Regression test parses a slash token, catches the error, then asserts a subsequent `parse_progression("C F G Am")` still returns root-first components.

### WR-01: Register mapping monotonically climbs to the top of the range

**Files modified:** `core/engine/loop_engine.py`, `tests/test_loop_engine.py`
**Commit:** 7b212be
**Applied fix:** `_choose_register_for_chord_tone` no longer short-circuits on closest-to-previous. The max-leap rule now only *narrows* the candidate set (candidates within 12 semitones of the previous note), then the register bias is applied inside that set: root/fifth tones prefer octave ≤ 3, other tones prefer octaves 3–4. Regression tests assert a 16-bar line is non-monotonic and does not park at the range ceiling.

### WR-02: Duet-only preset through the progression path leaks internal "Rhythm is empty" error

**Files modified:** `core/engine/loop_engine.py`, `tests/test_loop_engine.py`
**Commit:** 7b212be
**Applied fix:** `build_progression_score` (and therefore `generate_variant_from_progression`) guards empty `preset.rhythm` up front with an actionable message naming the preset as duet-only and listing valid solo presets — mirroring the WR-06 guard from Phase 02 on the rhythm field.

### WR-03: Quality-alias normalization is exact-match only — `Cmaj7/G` fails confusingly

**Files modified:** `core/engine/progression.py`
**Commit:** 567e192
**Applied fix:** Resolved by the CR-01 slash rejection: `Cmaj7/G` now fails with the explicit slash-not-supported error instead of the misleading "Unknown quality maj7".

### IN-01: `chord_tones_used` trace field has inconsistent semantics between generation paths

**Files modified:** `core/models.py`
**Commit:** afaeb4f
**Applied fix:** Documented on `GenerationTrace.chord_tones_used` that element shape differs per `pattern_strategy` (concrete octave-bearing pitches for `preset_verbatim`, octave-less pitch classes for `progression_driven_register_mapped`), so downstream consumers branch on strategy instead of assuming one shape.

### IN-02: pychord's enharmonic spellings leak into generated notation (Gm renders A#, not Bb)

**Files modified:** `core/engine/loop_engine.py`, `tests/test_loop_engine.py`
**Commit:** afaeb4f
**Applied fix:** Chosen pitches in the progression path are respelled to the preset key's flat/sharp preference via `getEnharmonic` (MIDI-preserving), so a flat-key line reads Bb instead of A#. Regression test covers flat-key respelling end-to-end.

### IN-03: Fifth detection assumes triadic ordering — power chords misclassified

**Files modified:** `core/engine/loop_engine.py`, `tests/test_loop_engine.py`
**Commit:** afaeb4f
**Applied fix:** The chord fifth is now detected by interval from the root (6/7/8 semitones mod 12) instead of assuming index 2, so power chords (`C5` → `['C', 'G']`) get the correct root/fifth low-register bias. Regression test covers power-chord handling.

### IN-04: Test suite never exercises flats or slash chords through the engine

**Files modified:** `tests/test_progression.py`, `tests/test_loop_engine.py`
**Commit:** 7b212be (plus tests in 567e192, afaeb4f)
**Applied fix:** Added regression tests for slash-rejection cache safety, a flat-key progression end-to-end, the leap constraint over 16 bars, non-monotonic register movement, the duet-only guard, flat respelling, and power chords.
